# 签到分析后台页面设计

- 日期：2026-04-22
- 状态：已确认，待进入实现计划
- 适用范围：后台管理员侧签到分析能力

## 1. 背景与目标

当前签到记录功能仅提供后台明细列表，适合做审计与单条查询，但不适合回答以下运营问题：

- 最近签到活跃度是否上升或下降
- 每日奖励发放是否平稳
- 奖励是否集中在某些金额区间
- 哪些用户签到最活跃

本次设计的目标是在**保留原有后台签到记录列表**的前提下，新增一个**后台侧边栏独立菜单**，提供签到趋势分析页面，帮助管理员先看整体趋势，再钻取到明细。

## 2. 已确认决策

### 2.1 页面范围

- 保留现有后台签到记录列表页，不替换、不合并
- 新增后台独立菜单页，名称暂定为“签到分析”
- 不给普通用户开放，不新增普通用户可见配置
- 新页面仅对管理员开放

### 2.2 视觉方向

- 采用方案 A：趋势看板
- 同时保留底部明细表，承担钻取职责
- 页面结构固定为：
  - 指标卡
  - 趋势图
  - 奖励分布 / Top 用户
  - 明细表

### 2.3 非目标

本次不包含以下内容：

- 普通用户查看签到分析
- 导出图表或导出分析报表
- 复杂图表联刷
- 时区对比分析
- 工作日 / 周末对比
- 改造现有签到规则或奖励算法

## 3. 信息架构

### 3.1 菜单与页面

- 现有页面：
  - `/admin/checkins`
  - 页面职责：签到明细审计
- 新增页面：
  - `/admin/checkin-analytics`
  - 页面职责：签到趋势分析

建议将新页面路由放在现有后台签到记录路由旁边，作为并列菜单项。

### 3.2 权限

前端路由与后端接口均沿用现有后台页的管理员权限模型：

- `requiresAuth: true`
- `requiresAdmin: true`

不新增额外权限层，不新增配置开关。

## 4. 页面布局设计

### 4.1 顶部筛选条

筛选条件统一作用于图表区和底部明细表。

建议包含：

- 时间范围快捷项：
  - 近 7 天
  - 近 30 天
  - 本月
  - 自定义
- 用户搜索：
  - 邮箱
  - 用户名
- 时区筛选
- 刷新按钮

默认时间范围：`近 30 天`

### 4.2 第一屏指标卡

建议展示 4 张卡片：

- 累计签到次数
- 累计发放奖励
- 今日签到人数
- 人均奖励

作用：

- 让管理员先快速了解总量与当天状态
- 为趋势图提供阅读背景

### 4.3 核心趋势区

主图为双指标趋势图：

- 主轴：每日签到次数
- 副轴：每日奖励总额

推荐图形：

- 折线图或面积折线图

推荐原因：

- 最适合表达时间趋势变化
- 能同时看出活跃度变化和奖励发放波动

### 4.4 下方分析区

左右两列布局：

- 左侧：奖励分布柱状图
- 右侧：Top 签到用户排行

#### 奖励分布柱状图

目标：

- 观察签到奖励集中在哪些金额区间
- 判断分档权重配置是否符合预期

#### Top 用户排行

默认排序：

- 按签到次数降序

展示字段：

- 用户名
- 邮箱
- 签到次数
- 累计奖励

### 4.5 底部明细表

底部保留一个明细表，继续承担审计和钻取功能。

建议复用现有表格能力：

- `DataTable`
- `Pagination`

职责：

- 响应图表联动后的筛选
- 展示最终明细

## 5. 交互设计

### 5.1 页面加载

- 页面首次进入时，使用默认筛选条件加载分析数据和明细数据
- 图表区与明细表同时刷新，但错误态彼此独立

### 5.2 筛选联动

当顶部筛选变化时：

- 趋势图刷新
- 指标卡刷新
- 奖励分布刷新
- Top 用户刷新
- 明细表刷新

所有模块共享同一套筛选口径。

### 5.3 趋势图下钻

点击趋势图上的某一天后：

- 底部明细表自动过滤到该日期
- 页面显示当前联动提示，例如：
  - `当前查看：2026-04-21`

### 5.4 Top 用户联动

点击 Top 用户排行的某一项后：

- 底部明细表自动过滤到该用户
- 搜索框同步显示该用户邮箱或用户名

### 5.5 清除联动

提供“清除联动”能力，使底部明细表恢复到当前时间范围下的全量结果。

### 5.6 不做的交互

第一版不做：

- 图表之间互相联动
- 多条件图表交叉钻取
- 图表导出
- 高级分析视图切换

## 6. 空态与异常态

### 6.1 空态

当筛选范围内无签到数据时：

- 指标卡显示 `0`
- 趋势图显示空态提示
- 奖励分布显示空态提示
- Top 用户显示空态提示
- 明细表显示空态提示

要求：

- 不出现空白区域
- 仅对无数据模块显示空态

### 6.2 错误态

当分析接口失败时：

- 图表区域显示错误信息
- 提供 `重试` 按钮

当明细接口失败时：

- 图表可继续保留
- 明细表单独显示错误态

不依赖仅 toast 的错误提示方式。

## 8. 后端接口设计

### 8.1 保留现有接口

继续保留：

- `GET /admin/checkins`

职责：

- 为底部明细表提供分页数据
- 保持与现有列表页兼容

### 8.2 新增聚合分析接口

建议新增：

- `GET /admin/checkins/analytics`

用途：

- 一次返回指标卡、趋势图、奖励分布、Top 用户排行所需的数据

### 8.3 查询参数

建议支持：

- `start_date`
- `end_date`
- `search`
- `timezone`
- `top_limit`

说明：

- `search` 用于邮箱或用户名模糊搜索
- `timezone` 用于按时区过滤
- `top_limit` 默认为 `10`

### 8.4 响应结构

```json
{
  "overview": {
    "total_checkins": 1280,
    "total_reward_amount": 18.5623,
    "today_checkins": 43,
    "avg_reward_amount": 0.0145
  },
  "trend": [
    {
      "date": "2026-04-01",
      "checkin_count": 38,
      "reward_amount": 0.53
    }
  ],
  "reward_distribution": [
    {
      "label": "0.002 - 0.005",
      "count": 120,
      "reward_amount": 0.46
    }
  ],
  "top_users": [
    {
      "user_id": 12,
      "user_name": "alice",
      "user_email": "alice@example.com",
      "checkin_count": 28,
      "reward_amount": 0.41
    }
  ]
}
```

对应关系：

- `overview`：顶部指标卡
- `trend`：趋势图
- `reward_distribution`：奖励分布图
- `top_users`：Top 用户排行

## 9. 服务层与仓储层设计

### 9.1 设计原则

- 列表查询与聚合分析分离
- 不将统计逻辑硬塞进现有分页接口
- 聚合分析接口单独建模

### 9.2 Service 层

建议新增：

- `GetAdminAnalytics(ctx, filters) (*AdminCheckinAnalytics, error)`

### 9.3 Repository 层

建议拆分为 4 类查询：

- `GetAdminOverview(...)`
- `GetAdminTrend(...)`
- `GetAdminRewardDistribution(...)`
- `GetAdminTopUsers(...)`

这样便于：

- 测试各统计口径
- 后续增量扩展分析维度
- 避免列表接口复杂化

## 10. 奖励分桶策略

奖励分布的金额分桶建议由后端统一计算，前端只负责展示。

推荐做法：

- 根据当前签到奖励区间，切分为固定 5 桶或 6 桶
- 由后端返回分桶标签和聚合结果

原因：

- 保证前后端口径一致
- 避免前端自行分桶导致显示与统计不一致
- 更利于测试和后续配置变化

## 11. 验证方案

### 11.1 后端验证

- Service 层：
  - 校验 overview 聚合口径
  - 校验 trend 时间范围聚合正确性
  - 校验 reward_distribution 分桶逻辑
  - 校验 top_users 排序正确性
- Handler 层：
  - 校验 `/admin/checkins/analytics` 参数解析
  - 校验响应结构
  - 校验管理员权限要求

### 11.2 前端验证

- 页面初始加载态
- 页面空态
- 页面错误态
- 筛选条件变化时图表和表格同时刷新
- 点击趋势点后明细表按日期过滤
- 点击排行项后明细表按用户过滤

### 11.3 回归重点

- 现有 `/admin/checkins` 页面行为不变
- 现有签到设置不发生行为回归
- 现有签到功能本身不受影响

## 12. 风险与权衡

### 12.1 风险

- 聚合查询可能比当前分页列表更重
- 时间范围、自定义筛选、图表联动会提升前后端状态复杂度
- 分桶策略若设计不稳，可能导致奖励分布图解释性不足

### 12.2 当前取舍

- 第一版只做管理员趋势看板，不扩展到普通用户
- 第一版只做单向钻取到明细表，不做复杂联动
- 第一版集中在趋势、分布、排行、明细 4 个核心模块

## 13. 最终方案摘要

本次最终方案为：

- 保留现有后台签到记录列表页
- 后台侧边栏新增独立菜单“签到分析”
- 页面结构为：
  - 指标卡
  - 趋势图
  - 奖励分布
  - Top 用户排行
  - 明细表
- 页面仅管理员可见
- 新增聚合接口 `GET /admin/checkins/analytics`
- 原有明细接口 `GET /admin/checkins` 保持不变

该设计已完成范围、权限、接口、交互和验证口径确认，可进入实现计划阶段。
