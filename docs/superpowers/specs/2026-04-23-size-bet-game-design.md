# 大小中竞猜游戏设计

- 日期：2026-04-23
- 状态：已确认，待进入实现计划
- 适用范围：当前项目内的首款运营型竞猜游戏

## 1. 背景与目标

当前项目已经具备用户体系、额度体系、签到奖励、后台配置和管理侧页面能力，适合作为运营小游戏的宿主系统。本次设计的目标是在**不拆分独立项目**的前提下，为当前项目新增一款“大小中竞猜”游戏，满足以下业务诉求：

- 用户使用当前账号直接参与
- 用户直接使用主站账号额度下注
- 游戏支持真实同池多人同局
- 页面具备明显倒计时、庄家区和玩家区视觉效果
- 平台通过公开概率与公开赔率实现长期正收益
- 每一笔下注、返奖、退款都可追溯，便于审计和排查刷额度问题
- 首发支持总榜和周榜

## 2. 已确认决策

### 2.1 产品定位

- 在当前项目内实现，不拆独立站点，不拆独立服务
- 首发只做一款游戏，但底层结构要为后续扩展第二款游戏留出空间
- 游戏属于运营活动模块，而不是实时游戏服务器

### 2.2 核心玩法

- 每局 60 秒
- 0 到 49 秒为下注阶段
- 50 到 59 秒为封盘等待阶段
- 整点开奖并进入下一期
- 所有在线玩家共享同一局、同一个开奖结果
- 每个账号每局只能下注 1 次
- 每次下注只能选择一个方向：`小 / 中 / 大`
- 每次下注额度从后台配置的额度列表中选择，首发默认值为：`2 / 5 / 10 / 20`

### 2.3 结果规则

- 开奖数字范围：`1-11`
- `1-5 = 小`
- `6 = 中`
- `7-11 = 大`
- `小 / 中 / 大` 的概率由后台配置
- 前端页面必须展示精确概率

### 2.4 赔率规则

- 后台可配置 `小 / 中 / 大` 三档赔率
- 默认赔率为：
  - `小 = 2.0x`
  - `中 = 10.0x`
  - `大 = 2.0x`
- 赔率为含本金返还的总返奖倍数
- 示例：
  - 押 `2` 中 `小`，返 `4`
  - 押 `5` 中 `大`，返 `10`
  - 押 `2` 中 `中`，返 `20`

### 2.5 平台收益原则

- 不允许根据玩家下注情况反推开奖结果
- 不允许“每局必赚”的暗箱控盘
- 平台盈利方式为：公开概率 + 公开赔率带来的长期正收益
- 概率与赔率必须在规则区对用户明示

### 2.6 开关与规则展示

- 仅保留 `游戏总开关`
- 游戏关闭时，页面仍可查看规则和历史结果，但不能下注
- 页面必须完整展示规则、概率、赔率、局时长、封盘规则、下注限制和异常退款规则

### 2.7 排行榜

- 首发支持：
  - 总榜
  - 周榜
- 推荐主指标：累计净盈利额度

## 3. 非目标

本次不包含以下内容：

- 实时 PvP 或多人即时对战
- 每秒同步战斗状态
- 复杂活动编排系统
- 多款游戏共用的完整游戏平台
- 独立游戏币体系
- 后台“本期暂停下注”单独开关
- 基于下注数据动态调整开奖结果

## 4. 用户可见规则设计

### 4.1 基础规则

- 游戏以期为单位运行
- 每期持续 60 秒
- 前 50 秒允许下注
- 后 10 秒封盘并等待开奖
- 每个账号每期只能下注 1 次
- 每次下注只能押一个方向
- 下注后不可修改、不可撤销

### 4.2 概率与赔率展示

- 页面首屏展示当前：
  - 小的概率与赔率
  - 中的概率与赔率
  - 大的概率与赔率
- 概率使用精确百分比展示，例如：
  - `小 45.00%`
  - `中 10.00%`
  - `大 45.00%`

### 4.3 开奖公平性说明

- 每期使用独立配置快照
- 配置快照在本期开启后固定，不受后台后续修改影响
- 每期开始时生成开奖承诺值 `server_seed_hash`
- 开奖后公开对应 `server_seed`
- 前端展示承诺值与开奖结果说明，便于解释“结果不是事后改的”

### 4.4 额度说明

- 用户直接使用主站账号额度参与
- 下注时先扣额度
- 中奖时按赔率返奖
- 未中奖则不返
- 如发生系统异常导致本期作废，整期下注按规则退款

## 5. 前端页面设计

建议将页面做成强活动感单页，而不是普通管理后台风格。

### 5.1 页面布局

页面分为以下 6 个区块：

- 顶部状态区
- 庄家区
- 玩家下注区
- 开奖结果区
- 排行榜与历史区
- 规则完整展示区

### 5.2 顶部状态区

展示：

- 游戏标题
- 当前期号
- 当前阶段：`下注中 / 封盘中 / 开奖中 / 维护中`
- 大号倒计时数字
- 进度条或环形倒计时效果

视觉要求：

- 下注阶段使用活跃色
- 封盘阶段切换为警示色
- 开奖阶段具备明显动画反馈

### 5.3 庄家区

展示：

- 庄家区标题
- 当前规则摘要
- 当前概率
- 当前赔率
- 当前期承诺值摘要
- 上期开奖结果入口

视觉要求：

- 与玩家下注区明显区分
- 采用更稳重的深色或金属感面板样式

### 5.4 玩家下注区

展示：

- 三个方向卡片：`小 / 中 / 大`
- 每个卡片展示：
  - 名称
  - 数字范围
  - 当前概率
  - 当前赔率
- 下注额度按钮组：`2 / 5 / 10 / 20`
- 确认下注按钮
- 本局我的下注状态

交互要求：

- 玩家先选方向，再选额度
- 成功下注后，本局下注区锁定
- 余额不足时禁用下注并明确提示
- 封盘后全部下注交互禁用

### 5.5 开奖结果区

展示：

- 当前开奖数字
- 所属分类：`小 / 中 / 大`
- 我的本局结果：中奖或未中奖
- 返奖额度或净输赢

动效要求：

- 开奖前有短动画滚动或翻牌效果
- 结果出现后高亮命中分类
- 如果用户中奖，展示余额变动反馈

### 5.6 排行榜与历史区

排行榜使用 Tab：

- 总榜
- 周榜
- 我的记录

我的记录至少展示：

- 期号
- 下注方向
- 下注额度
- 开奖数字
- 开奖分类
- 返奖额度
- 净输赢
- 时间

### 5.7 规则展示区

页面底部必须完整展示：

- 玩法说明
- 开奖数字范围
- 小中大的映射关系
- 当前精确概率
- 当前赔率
- 下注时间与封盘规则
- 单账号单局下注限制
- 额度来源说明
- 异常期退款说明
- 公平性说明

## 6. 后台配置设计

建议新增后台“运营游戏”或“竞猜游戏”入口。

### 6.1 基础配置

- 游戏总开关
- 游戏标题
- 游戏说明文案
- 局时长，默认 60 秒
- 封盘秒数，默认 50 秒
- 可下注额度列表

### 6.2 赔率与概率配置

- 小赔率
- 中赔率
- 大赔率
- 小概率
- 中概率
- 大概率
- 保存前对概率和赔率做校验
- 后台实时展示前端可见概率百分比预览

### 6.3 运行状态

- 当前期号
- 当前阶段
- 当前期承诺哈希
- 最近期开奖结果
- 是否存在异常未结算期次

### 6.4 审计与风控

- 按期号查看全部下注
- 按用户查看下注与流水
- 查某天所有返奖
- 异常高频下注筛查
- 重复请求或重复结算告警

## 7. 开奖逻辑设计

### 7.1 期次模型

- 每期生成唯一 `round_id`
- 每期开始时生成当期配置快照
- 快照包括：
  - 概率
  - 赔率
  - 额度列表
  - 时间窗口

### 7.2 概率抽样

开奖采用两段式：

1. 按 `小 / 中 / 大` 概率权重抽中类别
2. 在类别内部抽具体数字

数字分配规则：

- 小：在 `1-5` 内均匀抽取
- 中：固定为 `6`
- 大：在 `7-11` 内均匀抽取

### 7.3 配置冻结

- 本期开局后，快照固定
- 后台修改配置仅影响下一期
- 这样可以避免同一期前后展示规则不一致

## 9. 额度结算与审计设计

### 9.1 下注

用户下注成功时，事务内完成：

1. 写 `game_bets`
2. 写 `game_wallet_ledger` 的 `bet_debit`
3. 扣减 `users.balance`

### 9.2 开奖结算

开奖后对中奖用户，事务内完成：

1. 更新 `game_bets` 结算结果
2. 写 `game_wallet_ledger` 的 `bet_payout`
3. 增加 `users.balance`
4. 更新榜单聚合或快照

### 9.3 异常退款

若某期异常作废，事务内完成：

1. 更新下注记录状态为退款
2. 写 `game_wallet_ledger` 的 `bet_refund`
3. 退回用户余额

### 9.4 审计要求

必须保证以下问题可追溯：

- 用户在哪一期押了什么方向
- 押了多少额度
- 当期的概率与赔率是什么
- 最终开了什么数字
- 余额在扣款前后是多少
- 返奖或退款是多少
- 是否发生重复结算

## 10. 接口设计

建议新增独立命名空间：`/api/v1/game/size-bet`

### 10.1 用户侧接口

- `GET /current`
  - 获取当前期信息、倒计时、阶段、概率快照、赔率快照、我的下注状态、上期开奖结果摘要
- `POST /bet`
  - 提交下注
  - 入参：`round_id`、`direction`、`stake_amount`、`idempotency_key`
- `GET /history`
  - 获取我的下注历史
- `GET /rounds`
  - 获取最近若干期开奖结果
- `GET /leaderboard?scope=weekly|all`
  - 获取周榜或总榜
- `GET /rules`
  - 获取规则文案和当前公开配置

### 10.2 后台接口

- `GET /api/v1/admin/game/size-bet/settings`
- `PUT /api/v1/admin/game/size-bet/settings`
- `GET /api/v1/admin/game/size-bet/rounds`
- `GET /api/v1/admin/game/size-bet/bets`
- `GET /api/v1/admin/game/size-bet/ledger`
- `POST /api/v1/admin/game/size-bet/rounds/:id/refund`

## 11. 排行榜设计

### 11.1 榜单范围

- 总榜：统计上线以来累计净盈利
- 周榜：按自然周统计净盈利

### 11.2 主指标

首发主指标使用：`累计净盈利额度`

辅助字段：

- 胜场数
- 总参与局数
- 命中率

### 11.3 展示要求

- 用户名脱敏或使用昵称
- 支持刷新时间展示
- 被风控标记的账号可隐藏或剔除

## 12. 异常处理设计

### 12.1 下注重复提交

- 使用 `idempotency_key` 保障幂等
- 同一用户同一期重复请求只允许成功一次

### 12.2 开奖异常

- 若开奖任务异常，本期标记为异常期
- 已下注用户整期退款
- 前端显示“本期异常，已退款”

### 12.3 返奖失败

- 不允许只开奖不返奖
- 结算必须事务化或进入可恢复补偿流程

### 12.4 配置变更跨期

- 后台配置修改只影响下一期
- 当前期继续使用开局快照

### 12.5 游戏关闭

- 不允许新下注
- 已存在但未完成的期次按设计规则继续结算或整期退款

## 13. 服务端压力与可行性判断

按日活几百的规模，这款游戏属于中低压力、可控负载。

原因：

- 非实时对战
- 非逐秒状态同步
- 主要是下注、开奖、结算、榜单查询等低频业务请求
- 数据库压力主要来自事务写入，而不是高 CPU 计算

为了保持压力可控，建议：

- 排行榜使用缓存或快照
- 历史结果分页查询
- 避免频繁轮询
- 不做逐秒自动存档

## 14. 验收标准

- 用户可看到完整规则、赔率、概率、期号和倒计时
- 倒计时和封盘状态在页面上有明显视觉反馈
- 每个账号每局只能下注一次
- 超过 50 秒后不能下注
- 所有玩家共享同一期开奖结果
- 下注扣款、中奖返奖、异常退款都有独立流水可查
- 后台可修改总开关、赔率、概率和额度列表
- 每期都有独立配置快照
- 总榜和周榜可正常展示
- 历史记录可回溯到下注、开奖和账本明细
